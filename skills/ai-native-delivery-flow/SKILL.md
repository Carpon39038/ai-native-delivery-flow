# AI Native 研发交付工作流

AI Native 团队从需求到发布的完整交付流程，集成飞书（lark-cli）实现文档获取和任务同步。

## 核心流程

```
飞书产品需求 → 1. 开发需求文档 → [人工审核] → 2. 开发设计文档 + 测试清单 → [人工审核] → 3. 任务拆分清单 + 同步飞书任务 → 4. 开发执行 + 自动代码审核 + MR人工审核 → 5. 发布确认/发布清单

（文档回补贯穿全程，每次对话发现不一致立即更新）
```

## 前置依赖

需要安装 [lark-cli](https://github.com/larksuite/cli)：

```bash
npx @larksuite/cli@latest install
lark-cli config init
lark-cli auth login --recommend
```

## 核心原则

1. **产品需求可以轻，开发需求必须准** — 产品需求从飞书获取，允许不完整；开发需求是进入开发的正式契约，必须明确。
2. **开发需求和开发设计必须分开** — "做什么"和"怎么做"是不同的文档，由不同的人审核。
3. **AI 可以辅助生成，关键节点必须人工审核** — AI 负责生成初稿、预审代码；人工必须确认：开发需求定稿、开发设计定稿、MR 审核通过、发布确认。
4. **文档回补贯穿全程** — 任何阶段发现需求缺口、设计变更、测试遗漏，立即更新对应文档，不等到最后。
5. **交付不是代码合并，而是发布后可观测闭环** — 完整交付包含发布确认、监控重点和回滚方案。

## 文档目录结构

```
requirements/<功能名>/
  01_product_requirement.md    # 飞书原始产品需求
  02_dev_requirement.md        # 开发需求文档
  03_dev_design.md             # 开发设计文档
  04_test_checklist.md         # 测试清单
  05_task_breakdown.md         # 任务拆分清单
  06_release_checklist.md      # 发布确认清单
```

## 文档回补规则（贯穿全程）

在任何阶段的对话过程中，如果发现以下情况，必须立即更新对应文档：

- 需求边界有变化 → 更新 `02_dev_requirement.md`
- 接口/字段/方案有变化 → 更新 `03_dev_design.md`
- 发现新的测试场景 → 更新 `04_test_checklist.md`
- 任务范围有调整 → 更新 `05_task_breakdown.md`

**做法：** 每次对话中如果讨论到与已有文档不一致的内容，先确认变更，然后直接修改文档并在变更记录中注明原因和日期。

## 步骤一：开发需求文档

### 输入

用户提供飞书产品需求文档链接。

### 执行流程

**1. 从飞书获取产品需求**

```bash
lark-cli docs +fetch --api-version v2 --doc "<飞书文档URL或token>" --scope body --detail rich
```

如果是知识库链接（`.../wiki/<token>`）：

```bash
lark-cli wiki spaces get_node --params '{"token":"<wiki_token>"}'
lark-cli docs +fetch --api-version v2 --doc "<obj_token>" --scope body --detail rich
```

将原始内容保存为 `01_product_requirement.md`，标注来源和获取时间。

认证失败时引导用户执行 `lark-cli auth login --recommend`。

**2. 生成开发需求初稿**

基于产品需求内容，结合仓库代码上下文（现有模块结构、接口风格、权限模型），生成 `02_dev_requirement.md`。

生成时：
- 从产品需求中提取已有的明确信息
- 补充开发视角：功能范围、不做范围、页面状态、数据规则、权限规则、边界情况、异常情况
- 产品需求中未明确的部分标记为"⚠️ 待确认"，不要自行猜测
- 区分"来源：产品需求"和"来源：AI 推导"

**3. 交用户审核**

向用户展示完整的开发需求文档，附带审核清单、AI 发现的缺口列表和所有"待确认问题"。确认通过后才进入步骤二。

### 开发需求文档模板

```markdown
# 开发需求：<功能名>

> 产品需求来源：<飞书文档链接>
> 创建日期：<日期>

## 背景

## 目标

## 功能范围
- [ ] 功能点 1
- [ ] 功能点 2

## 不做范围
- 功能 X（原因）

## 用户场景
1. 场景 A：...

## 交互规则

## 页面状态
| 状态 | 说明 |
|------|------|
| 加载中 | ... |
| 空数据 | ... |
| 正常 | ... |
| 错误 | ... |

## 数据规则

## 权限规则

## 边界情况

## 异常情况

## 验收标准
- [ ] 标准 1
- [ ] 标准 2

## 待确认问题

## 变更记录
| 日期 | 变更内容 | 来源 |
|------|----------|------|
```

### 审核清单

- [ ] 功能范围是否明确？
- [ ] 不做范围是否明确？
- [ ] 页面状态是否完整？
- [ ] 数据规则是否完整？
- [ ] 权限规则是否明确？
- [ ] 边界情况是否覆盖？
- [ ] 异常情况是否覆盖？
- [ ] 验收标准是否可执行？
- [ ] 待确认问题是否已全部解决？

## 步骤二：开发设计文档 + 测试清单

### 前置条件

步骤一的开发需求已通过用户审核。

### 执行流程

同时生成两个文档：
1. `03_dev_design.md` — 工程方案
2. `04_test_checklist.md` — 测试清单

生成时需要：
- 读取已确认的 `02_dev_requirement.md`
- 读取项目代码了解现有架构、接口风格、组件结构
- 设计文档要具体到接口、字段、模块结构
- 测试清单要覆盖主路径、空数据、边界、异常、权限、兼容

生成后交用户审核，确认通过后进入步骤三。

### 开发设计文档模板

```markdown
# 开发设计：<功能名>

## 方案概述

## 接口设计
// 接口地址、请求/响应类型（TypeScript）

## 字段模型
// 类型定义（TypeScript）

## 模块结构

## 状态流转
1. 用户操作 → 2. 状态更新 → 3. 接口请求 → 4. 重新渲染

## 重点变更

## 影响范围

## 兼容方案

## 风险提示
- [业务] ...
- [兼容] ...
- [性能] ...
- [数据] ...

## 发布注意事项

## 待确认问题

## 变更记录
| 日期 | 变更内容 | 来源 |
|------|----------|------|
```

### 测试清单模板

```markdown
# 测试清单：<功能名>

## 主路径测试
- [ ] ...

## 功能点测试
- [ ] ...

## 空数据测试
- [ ] ...

## 边界情况测试
- [ ] ...

## 异常情况测试
- [ ] ...

## 权限测试
- [ ] ...

## 兼容测试
- [ ] ...

## 性能关注点
- [ ] ...

## 回归测试点
- [ ] ...

## 暂未覆盖项
- ...（原因）
```

### 审核清单

**开发设计：**
- [ ] 方案是否满足开发需求？
- [ ] 接口设计是否完整？
- [ ] 字段模型是否清晰？
- [ ] 状态流转是否合理？
- [ ] 是否影响旧逻辑？
- [ ] 兼容方案是否明确？
- [ ] 风险是否识别充分？

**测试清单：**
- [ ] 主路径是否覆盖？
- [ ] 边界和异常是否覆盖？
- [ ] 权限测试是否包含？
- [ ] 暂未覆盖项是否有合理原因？

## 步骤三：任务拆分清单

### 前置条件

步骤二的开发设计和测试清单已通过用户审核。

### 执行流程

**1. 生成任务拆分**

基于开发设计和测试清单，将工作拆分为可独立执行和审查的小任务。生成 `05_task_breakdown.md`。

**2. 同步到飞书任务**

用户确认任务拆分后，使用 `lark-cli` 创建飞书任务：

```bash
lark-cli task create --params '{"task":{"summary":"<任务标题>","description":"<任务描述>"}}'
lark-cli task tasklists add_tasks --params '{"tasklist_guid":"<列表ID>","task_guids":["<任务ID>"]}'
```

### 任务模板

```markdown
## 任务 N：<标题>

**描述：** 一句话说明目标。

**修改范围：**
- file1.ts
- file2.tsx

**不修改范围：**
- fileX.ts（原因）

**验收点：**
- [ ] 检查项 1
- [ ] 检查项 2

**对应测试项：** 测试清单中的哪些项

**风险点：** ...
```

### 任务拆分原则

1. 每个任务目标单一
2. 每个任务有明确的修改范围和不修改范围
3. 每个任务能独立 review
4. 每个任务对应测试清单中的测试项
5. 不要在一个任务中包含无关改动

---

## 步骤四：开发执行

### 前置条件

步骤三的任务拆分已确认，飞书任务已同步。

### 执行约束

按任务逐个执行开发，每个任务严格遵守：

1. 不得擅自扩大需求范围
2. 不得修改任务范围外的文件
3. 不得随意新增依赖
4. 不得绕过权限逻辑
5. 不得破坏旧接口兼容
6. 发现缺口 → 先更新对应文档，再继续开发

### AI 指令模式

推荐：
```
请根据 03_dev_design.md，完成任务 N：<标题>。
只允许修改：<文件列表>。
不要修改：<排除列表>。
完成后说明修改内容、风险点和自测方式。
```

不推荐：
```
帮我实现这个功能
```

### 自动代码审核

每个任务完成后，AI 自动进行预审，检查：
- 是否超出任务范围
- 是否符合开发需求和开发设计
- 是否存在明显 bug
- 是否缺少空数据/异常处理
- 是否有性能/权限/数据风险
- 对应的测试项是否覆盖

### 提交 MR 人工审核

预审通过后，生成 MR 描述并提交，等待人工审核。

**MR 描述模板：**

```markdown
## 关联文档
- 开发需求：链接
- 开发设计：链接
- 任务：任务 N

## 本次改动
1. ...

## 修改范围 / 未修改范围

## 测试清单覆盖
- [x] 已覆盖项
- [ ] 未覆盖项（原因）

## 风险点

## 需要 Reviewer 重点关注
```

**MR 审查清单：**
- [ ] 是否符合开发需求？
- [ ] 是否符合开发设计？
- [ ] 是否只修改了任务允许范围？
- [ ] 是否有无关改动？
- [ ] 是否破坏旧逻辑？
- [ ] 是否遗漏权限/空数据/异常处理？
- [ ] 是否存在性能/兼容风险？
- [ ] 测试清单关键项是否覆盖？

### 飞书任务状态更新

任务完成并 MR 合并后，更新飞书任务状态：

```bash
lark-cli task patch --params '{"task_guid":"<任务ID>","update_fields":["status"]}' --data '{"task":{"status":"done"}}'
```

## 步骤五：发布确认 / 发布清单

### 前置条件

所有任务已完成，所有 MR 已合并。

### 执行流程

生成 `06_release_checklist.md`，包含发布前检查和发布后监控计划。

### 发布清单模板

```markdown
# 发布清单：<功能名>

## 发布内容
- 功能描述

## 关联文档
- 开发需求、开发设计、测试清单

## 发布前检查
- [ ] 构建通过
- [ ] 类型检查通过
- [ ] 核心测试通过
- [ ] 测试清单完成情况（已覆盖 / 未覆盖）
- [ ] 所有 MR 已合并
- [ ] 已知风险已记录

## 影响范围

## 已知风险

## 回滚方案

## 发布后监控重点
- 前端：JS 错误、接口错误、白屏、性能指标
- 后端：接口错误率、接口耗时、数据库错误、任务/队列失败

## 发布负责人

## 发布时间
```

发布清单生成后交用户确认，确认后执行发布。

## 交互方式

当用户触发此 skill 时：

1. 询问当前需要哪个步骤
2. 如果是"新功能"或提供了飞书文档链接 → 从步骤一开始
3. 如果 `requirements/` 下已有对应功能目录 → 从当前步骤继续

### 关键约束

- 每个步骤之间都有人工审核节点，未确认不能进入下一步
- 对话中发现与已有文档不一致的内容 → 立即更新文档
- `lark-cli` 认证失败 → 引导用户执行 `lark-cli auth login --recommend`
- 读取 `requirements/` 下已有文档作为上下文，每个步骤都建立在前面步骤的基础上
